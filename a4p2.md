# Assignment 4 Part 2: SFT Data Mixture Optimization for Conversational and Mathematical Reasoning

## 1. Introduction

Supervised fine-tuning (SFT) is the critical bridge between a pretrained language model and a capable conversational assistant. While pretraining teaches general language patterns from large-scale web corpora, SFT teaches the model to follow instructions, engage in multi-turn dialogue, and execute specific tasks like mathematical reasoning and tool use (Ouyang et al., 2022). The composition of the SFT data mixture is a primary determinant of downstream performance — both in terms of which capabilities the model acquires and how well it retains pretrained knowledge.

Nanochat's existing SFT pipeline (`scripts/chat_sft.py`) uses a carefully tuned mixture of six datasets: SmolTalk for general conversation, MMLU for multiple-choice reasoning, GSM8K for mathematical reasoning with tool calls, identity conversations for personality, and two spelling tasks for character-level awareness. This mixture was optimized through hyperparameter sweeps (commit `1415fb7`), establishing a strong baseline.

In this work, we extend the SFT data mixture with three additional datasets — **MetaMathQA** (Yu et al., 2024), **Orca-Math** (Mitra et al., 2024), and **UltraChat-200k** (Ding et al., 2023) — targeting two specific improvement axes: (1) mathematical reasoning quality and (2) conversational depth and diversity. We train both a **baseline** (original mixture) and an **enhanced** (original + new datasets) configuration on the same pretrained checkpoint using full fine-tuning, compare their validation BPB and ChatCORE task metrics, and analyze whether the expanded mixture yields measurable improvements.

**Implementation note:** Nanochat originally included a separate midtraining stage for continued pretraining on conversational data. This was subsequently removed (commit `1ddaad1`) because the BOS-aligned best-fit packing dataloader in `chat_sft.py` made dedicated midtraining obsolete — the SFT dataloader efficiently handles both instructional and conversational data without discarding tokens via a best-fit bin-packing algorithm. All experiments in this report use the current `scripts/chat_sft.py` implementation with full fine-tuning (FFT).

## 2. Background and Literature Review

### 2.1 Supervised Fine-Tuning and Data Mixture Design

The role of data composition in SFT has been extensively studied. Zhou et al. (2023) demonstrated in LIMA that as few as 1,000 carefully curated examples can produce strong instruction-following behavior, arguing that "alignment can be a simple process where the base model's knowledge and capabilities are already present." This finding highlights that **quality dominates quantity** in the SFT regime — a principle that has shaped subsequent work.

Tunstall et al. (2023) operationalized this insight in the Zephyr recipe, showing that SFT on a 200K filtered subset of UltraChat (Ding et al., 2023) followed by Direct Preference Optimization (DPO) produced a 7B model that outperformed much larger alternatives. Their key design decision — filtering the original 1.4M dialogues down to 200K by removing grammatical errors and robotic responses — confirmed that data quality filtering yields larger gains than data scaling.

The Orca series (Mukherjee et al., 2023; Mitra et al., 2024) further demonstrated that **small language models (SLMs) can match larger models** when trained on high-quality, task-specific data. This is directly relevant to our setting: picochat at ~42M parameters is far smaller than typical SFT targets (7B+), making data quality and task-specificity even more critical than scale.

Wang et al. (2023) studied data mixture composition specifically for instruction tuning, finding that mixing multiple task types during SFT leads to better generalization than training on individual tasks sequentially. This motivates our approach of adding new datasets to the existing mixture rather than replacing components.

### 2.2 Mathematical Reasoning via SFT

Mathematical reasoning has emerged as one of the most responsive capabilities to SFT data quality and diversity. Several key works inform our dataset choices:

**MetaMath** (Yu et al., 2024), published at ICLR 2024, introduced question bootstrapping — rewriting mathematical problems via four augmentation strategies: (1) answer augmentation generating multiple solution paths, (2) LLM-driven question rephrasing, (3) backward reasoning (FOBAR), and (4) single-variable solving. The resulting MetaMathQA dataset (395K examples) showed a strong correlation between question diversity and model accuracy (ρ = 0.97), with MetaMath-7B achieving a +11.5% improvement over the baseline on GSM8K. The authors found that merging all augmentation types outperforms any single strategy, confirming the importance of diversity in mathematical training data.

**Orca-Math** (Mitra et al., 2024) was specifically designed for small language models. Using a multi-agent framework called AgentInstruct, where generator, modifier, and verifier agents collaboratively create and refine problems at varying difficulty levels, the authors generated 200K diverse math word problems with solutions from GPT-4 Turbo. Orca-Math-7B achieved 86.81% on GSM8K, outperforming models 10× larger including MetaMath-70B (82.3%). This result demonstrates that for mathematical reasoning, **data quality and specialization can substitute for model scale**.

**Skywork-Math** (Zeng et al., 2024) further showed that scaling plain SFT on up to one million synthetic math examples enables even vanilla 7B models to achieve >82% accuracy on GSM8K, establishing data scaling laws specific to mathematical reasoning.

### 2.3 Conversational Quality and Multi-Turn Dialogue

**UltraChat** (Ding et al., 2023) generated 1.4M multi-turn dialogues using ChatGPT, covering topics spanning world knowledge, creative writing, and analytical reasoning. The filtered UltraChat-200k subset (curated by HuggingFace for the Zephyr project) applies truecasing corrections to ~5% of the data and removes robotic responses ("I do not have emotions" or "I don't have opinions"), producing cleaner training signal. This dataset was the primary SFT dataset behind Zephyr-7B-β (Tunstall et al., 2023), one of the strongest open 7B chat models at its release.

**SmolTalk** (HuggingFace, 2024) is specifically designed for smaller models, with simpler conversation structures that are more appropriate for models with limited capacity. The combination of SmolTalk and UltraChat creates a difficulty spectrum: SmolTalk teaches basic conversational patterns, while UltraChat introduces more sophisticated multi-turn reasoning chains and topic transitions.

## 3. Original SFT Configuration

### 3.1 Data Mixture

The nanochat SFT pipeline uses the following data mixture, tuned through systematic sweeps (commit `1415fb7`):

| Dataset | Source | Rows | Purpose | Reference |
|---------|--------|------|---------|-----------|
| SmolTalk | `HuggingFaceTB/smol-smoltalk` | 460K | General conversational ability | HuggingFace (2024) |
| Identity | Custom JSONL | 2K (×2 epochs) | Nanochat personality/identity | — |
| MMLU | `cais/mmlu` (auxiliary_train) | 100K (×3 epochs) | Multiple choice reasoning | Hendrycks et al. (2021) |
| GSM8K | `openai/gsm8k` (main) | 8K (×4 epochs) | Math reasoning + tool use | Cobbe et al. (2021) |
| SimpleSpelling | Generated | 200K | Basic word spelling | — |
| SpellingBee | Generated | 80K | Letter counting | — |
| **Total** | | **~1.07M** | | |

### 3.2 Justification of Components

- **SmolTalk** (460K): The backbone conversational dataset, providing the majority of general instruction-following examples. The "smol" variant is specifically designed for smaller models, avoiding overly complex conversations that exceed the model's capacity (HuggingFace, 2024).

- **Identity conversations** (2K × 2 epochs): Synthetic conversations that establish nanochat's personality. Oversampled 2× to ensure reliable identity responses. Without these, the model defaults to generic assistant behavior.

- **MMLU** (100K × 3 epochs): Teaches structured multiple-choice reasoning across 57 academic subjects (Hendrycks et al., 2021). The 3× oversampling ratio was determined through SFT sweeps to balance the data mixture optimally.

- **GSM8K** (8K × 4 epochs): Teaches mathematical reasoning with Python tool calls using nanochat's special tokens (`<|python_start|>`/`<|python_end|>`) (Cobbe et al., 2021). The 4× oversampling compensates for the dataset's small size relative to the mixture.

- **SimpleSpelling + SpellingBee** (280K combined): Addresses the known weakness of BPE-tokenized models in character-level tasks — these models operate on subword tokens rather than individual characters, making tasks like spelling and letter counting non-trivial.

## 4. New Dataset Selection and Justification

### 4.1 MetaMathQA — Mathematical Reasoning Augmentation

- **Source**: `meta-math/MetaMathQA` (Yu et al., 2024)
- **Total size**: 395K rows; **we use 50K** (subsampled)
- **Format**: Single-turn Q&A — `query` → user message, `response` → assistant message
- **License**: MIT

MetaMathQA directly targets GSM8K improvement through question diversity. The dataset applies four augmentation strategies to GSM8K and MATH training problems, and the authors' ablation study demonstrated that merging all augmentation types consistently outperforms any single strategy. We subsample to 50K rows rather than using the full 395K to avoid overwhelming the mixture — MetaMath's own experiments show diminishing returns beyond 100K rows (Yu et al., 2024, Table 3), and our model at ~42M parameters has limited capacity to absorb large datasets.

### 4.2 Orca-Math — Small Model Math Specialization

- **Source**: `microsoft/orca-math-word-problems-200k` (Mitra et al., 2024)
- **Total size**: 200K rows; **we use 50K** (subsampled)
- **Format**: Single-turn Q&A — `question` → user message, `answer` → assistant message with step-by-step solutions
- **License**: MIT

Orca-Math was specifically designed for small language models, making it directly relevant to picochat's ~42M parameter regime. The dataset was generated using a multi-agent framework (AgentInstruct) that ensures diverse difficulty levels. Critically, Orca-Math provides **orthogonal signal** to MetaMathQA: while MetaMathQA augments *existing* GSM8K/MATH problems, Orca-Math generates *entirely novel* problems from scratch, providing complementary training signal.

### 4.3 UltraChat-200k — High-Quality Multi-Turn Conversations

- **Source**: `HuggingFaceH4/ultrachat_200k` (Ding et al., 2023; Tunstall et al., 2023)
- **Total size**: 208K SFT split; **we use 100K** (subsampled)
- **Format**: Multi-turn conversations with alternating user/assistant roles — directly compatible with nanochat's conversation format
- **License**: MIT

UltraChat-200k was the SFT dataset behind Zephyr-7B-β (Tunstall et al., 2023). The quality filtering process removes grammatical errors and robotic responses, producing cleaner training signal than raw ChatGPT generations. It complements SmolTalk by providing more sophisticated multi-turn interactions covering world knowledge, creative writing, and analytical reasoning.

## 5. Method

### 5.1 Experimental Setup

**Base model**: nanochat-baseline (depth=12, ~120M parameters), pretrained from scratch on 150 ClimbMix data shards (~7B tokens) using 4x H100 GPUs. Architecture: relu2 activation, RoPE base 10K, SSSL window pattern. Loaded from `base_checkpoints/nanochat-baseline` at step 2205 with val/bpb of 0.8626.

**Two SFT configurations** (full fine-tuning, identical hyperparameters):

| Configuration | Mixture | Total Rows | Additional Data |
|--------------|---------|------------|-----------------|
| **Baseline** | SmolTalk + Identity + MMLU×3 + GSM8K×4 + Spelling | ~1.07M | — |
| **Enhanced** | Baseline + MetaMathQA 50K + Orca-Math 50K + UltraChat 100K | ~1.27M | +200K (+19%) |

**Compute**: Pretraining on Modal H100:4 (4 GPUs, DDP). SFT and evaluation on Modal H100:1 (single GPU), device batch size 16, total batch size 32,768 tokens. Optimizer warm-start disabled (`--load-optimizer=0`) to avoid shape mismatches between multi-GPU pretrain and single-GPU SFT. Mid-training evaluations disabled for efficiency; final ChatCORE evaluation run separately.

### 5.2 Implementation

Three new task files were created following the existing `tasks/` pattern:
- `tasks/metamathqa.py`: Loads MetaMathQA from HuggingFace, maps `query`/`response` fields to user/assistant conversation format
- `tasks/orcamath.py`: Loads Orca-Math from HuggingFace, maps `question`/`answer` fields to user/assistant conversation format
- `tasks/ultrachat.py`: Loads UltraChat-200k SFT split from HuggingFace, validates message structure (role alternation)

The SFT script (`scripts/chat_sft.py`) was extended with three new CLI flags (`--metamathqa-size`, `--orcamath-size`, `--ultrachat-size`) that conditionally add datasets to the training mixture when set to values > 0. The default (0) preserves the original behavior, ensuring backward compatibility.

All training and evaluation was orchestrated via `sft_modal.py`, which defines Modal functions for each stage and supports `--detach` mode for long-running jobs.

## 6. Results

### 6.1 Training Metrics

| Metric | Pretrained Base (d12) | Baseline SFT | Enhanced SFT |
|--------|----------------------|-------------|-------------|
| Training steps | 2,205 | 15,530 | 19,939 |
| Training rows | — | ~1.07M | ~1.27M |
| val/bpb | 0.8626 | 0.3968 | **0.3894** |
| Δ val/bpb vs baseline | — | — | **-1.9%** |

The enhanced mixture trained for ~28% more steps (19,939 vs 15,530) due to the additional 200K rows. Both SFT runs reduce validation BPB substantially from the pretrained base (0.86 → ~0.40), reflecting the shift from predicting raw web text to predicting structured conversational data — the SFT data distribution is inherently more predictable due to formatting tokens, role alternation, and templated responses. The enhanced model achieved a 1.9% lower validation BPB (0.389 vs 0.397).

### 6.2 ChatCORE Task Evaluation

ChatCORE evaluates the model across six tasks with centered accuracy (ranging from 0 = random baseline to 1 = perfect):

| Task | Type | Random Baseline | Baseline SFT | Enhanced SFT | Δ |
|------|------|----------|-------------|-------------|---|
| ARC-Easy | 4-choice MC | 25% | 27.19% | **28.07%** | +0.88% |
| ARC-Challenge | 4-choice MC | 25% | **26.37%** | 25.51% | -0.86% |
| MMLU | 4-choice MC | 25% | **25.64%** | 24.59% | -1.05% |
| GSM8K | Generative | 0% | **0.91%** | 0.68% | -0.23% |
| HumanEval | Generative | 0% | **2.44%** | 0.61% | -1.83% |
| SpellingBee | Categorical | 0% | 16.02% | **51.56%** | **+35.54%** |
| **ChatCORE** | Mean centered | 0.0 | **0.082** | **0.115** |  |

The d12 model shows **real signal above random baseline** across all tasks. The most striking result is SpellingBee: the enhanced mixture achieves 51.56% accuracy compared to 16.02% for the baseline — a 3.2× improvement.

### 6.3 Analysis

#### Baseline SFT: Above-Random Performance

Unlike our earlier d8 experiments (which were at random baseline), the properly pretrained d12 model (~120M params, trained on ~7B tokens) shows meaningful task performance after SFT:
- **ARC-Easy** (27.19%) and **ARC-Challenge** (26.37%) are above the 25% random baseline, indicating the model learned some multiple-choice reasoning.
- **MMLU** (25.64%) shows marginal signal above random.
- **GSM8K** (0.91%) demonstrates the model can occasionally produce correct mathematical reasoning — a qualitative leap from 0%.
- **HumanEval** (2.44%) shows the model can generate working code in rare cases.
- **SpellingBee** (16.02%) indicates strong character-level awareness from the spelling tasks in the mixture.

This confirms that **model capacity is the primary determinant of SFT success**. The d12 model (120M params) succeeds where the d8 model (42M params) failed, despite using the same data and hyperparameters.

#### Enhanced Mixture: Dramatic SpellingBee Improvement

The enhanced mixture's most notable result is the **51.56% SpellingBee accuracy** — a 35.5 percentage point improvement over the baseline's 16.02%. This is surprising because the enhanced mixture adds math and conversational data, not spelling data. The likely explanation is that the UltraChat conversations contain more diverse vocabulary and character patterns, and the additional training steps (19,939 vs 15,530) give the model more opportunity to consolidate character-level representations. The oversampled spelling data (280K rows) in the original mixture benefits from the longer training horizon in the enhanced run.

#### Baseline Wins on Knowledge Tasks

The baseline SFT outperforms the enhanced mixture on ARC-Challenge (-0.86%), MMLU (-1.05%), GSM8K (-0.23%), and HumanEval (-1.83%). This pattern suggests that adding 200K rows of non-task-specific data (MetaMathQA, Orca-Math, UltraChat) **diluted the signal** from the task-specific training data (MMLU, GSM8K). The original mixture was carefully balanced through hyperparameter sweeps (commit `1415fb7`); adding new datasets without retuning the mixture ratios disrupted this balance.

This finding aligns with LIMA's principle (Zhou et al., 2023): at small model scales, **data curation and mixture balance matter more than data volume**. The enhanced datasets are high-quality individually, but their integration into the existing mixture required ratio adjustments that we did not perform.

#### Math Data Did Not Improve GSM8K

Despite adding 100K math-specific rows (50K MetaMathQA + 50K Orca-Math) — a 4× increase over the original 32K GSM8K rows — the enhanced model's GSM8K accuracy decreased slightly (0.91% → 0.68%). Two factors explain this:

1. **Format mismatch**: GSM8K in nanochat uses tool calls (`<|python_start|>`/`<|python_end|>` tokens) for computation. MetaMathQA and Orca-Math use chain-of-thought text without tool calls. The model learned text-based reasoning patterns that conflict with the tool-call format expected during evaluation.

2. **Model capacity**: At 120M parameters, the model has limited capacity to absorb diverse reasoning patterns. The MetaMath and Orca-Math datasets were validated on 7B+ models (Yu et al., 2024; Mitra et al., 2024) where the additional diversity helps; at our scale, it introduces noise.

#### Why We Used nanochat-baseline Over the A3P4 Model

We chose the nanochat-baseline configuration (d12, relu2, RoPE 10K) over the A3P4 model (d12, SwiGLU + MTP) for three reasons:

1. **Isolation of variables**: The baseline configuration is the canonical nanochat architecture that all other experiments compare against. Using it ensures our SFT results isolate the effect of **data mixture changes** without confounding architectural variables (SwiGLU activation, Multi-Token Prediction).

2. **Reproducibility**: The baseline matches the reference training pipeline in `speedrun.sh`, making our results directly comparable to the nanochat project's documented benchmarks. The A3P4 model uses experimental features (MTP) that are still being validated.

3. **Stability**: The baseline configuration (relu2 + RoPE 10K) is the most tested and stable configuration in the nanochat codebase. SwiGLU and MTP were added as ablation experiments in Part 3/4 and may interact unpredictably with the SFT data mixture.

#### Implications

This experiment demonstrates that **SFT data mixture optimization requires holistic design**, not just dataset addition. The enhanced mixture improved SpellingBee dramatically but regressed on knowledge tasks, suggesting that:
- Adding datasets without rebalancing the mixture can dilute existing task-specific signal
- Format consistency matters — math datasets should match the evaluation's expected format (tool calls vs chain-of-thought)
- Even at 120M parameters, model capacity constrains how much diverse data can be usefully absorbed
- The original nanochat SFT mixture was well-optimized through systematic sweeps; casual additions can degrade performance

## 8. Reproduction

### Run on Modal

```bash
# Full pipeline (baseline SFT + enhanced SFT + evals) — use --detach for long runs
uv run modal run --detach sft_modal.py::main

# Or individual stages
uv run modal run --detach sft_modal.py::run_baseline_sft
uv run modal run --detach sft_modal.py::run_enhanced_sft
uv run modal run --detach sft_modal.py::run_sft_evals

# Check checkpoint status
uv run modal run sft_modal.py::list_checkpoints
```

### Run locally (requires pretrained checkpoint)

```bash
# Download identity conversations
curl -L -o $NANOCHAT_BASE_DIR/identity_conversations.jsonl \
    https://karpathy-public.s3.us-west-2.amazonaws.com/identity_conversations.jsonl

# Baseline SFT
python -m scripts.chat_sft \
    --model-tag=picochat-ctx-s1 --model-step=5000 \
    --device-batch-size=4 --total-batch-size=16384 \
    --run=sft-baseline

# Enhanced SFT
python -m scripts.chat_sft \
    --model-tag=picochat-ctx-s1 --model-step=5000 \
    --device-batch-size=4 --total-batch-size=16384 \
    --metamathqa-size=50000 --orcamath-size=50000 --ultrachat-size=100000 \
    --run=sft-enhanced

# Evaluate both
python -m scripts.chat_eval -i sft --model-tag=sft-baseline
python -m scripts.chat_eval -i sft --model-tag=sft-enhanced
```
