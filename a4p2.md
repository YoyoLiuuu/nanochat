# Assignment 4 Part 2: SFT Data Mixture Optimization

## 1. Introduction

Supervised fine-tuning (SFT) transforms a pretrained base model into a conversational assistant by training on curated instruction-response pairs. The quality and composition of the SFT data mixture directly determines the model's conversational ability and task performance. Nanochat's existing SFT pipeline uses a mixture of six datasets targeting conversation, multiple choice reasoning, math with tool use, identity, and spelling tasks.

This experiment extends the SFT data mixture with three additional datasets — MetaMathQA, Orca-Math, and UltraChat-200k — chosen to improve mathematical reasoning and conversational quality. We run two SFT configurations on the same pretrained checkpoint, compare their ChatCORE metrics and individual task accuracies, and analyze whether the expanded mixture improves performance.

**SFT implementation note:** Nanochat originally had a separate midtraining stage for continued pretraining on conversational data. This was removed (commit `1ddaad1`) because the BOS-aligned best-fit packing dataloader in `chat_sft.py` made dedicated midtraining obsolete — the SFT dataloader handles both instruction and conversational data efficiently without discarding tokens. All experiments use the current `scripts/chat_sft.py` implementation.

## 2. Background and Literature Review

### 2.1 SFT Data Mixture Design

The composition of the SFT data mixture is a critical design decision. LIMA (Zhou et al., 2023) demonstrated that as few as 1,000 carefully curated examples can produce strong instruction-following behavior, arguing that "alignment can be a simple process where the base model's knowledge and capabilities are already present." However, subsequent work has shown that for task-specific capabilities like math reasoning and tool use, additional targeted data is essential.

The Zephyr recipe (Tunstall et al., 2023) established a practical template: SFT on UltraChat-200k followed by DPO alignment. Their key insight was that **data quality matters more than quantity** — the 200K filtered subset of UltraChat outperformed training on the full 1.4M dialogues. This principle guided our subset sizes: we use 50K-100K rows per new dataset rather than the full datasets.

The Orca papers (Mukherjee et al., 2023; Mitra et al., 2024) showed that small language models (SLMs) can match larger models when trained on high-quality, task-specific data. Orca-Math achieved 86.81% on GSM8K with a 7B model, outperforming MetaMath-70B — demonstrating that data quality and specialization can substitute for model scale.

### 2.2 Math Reasoning via SFT

Mathematical reasoning is one of the most responsive capabilities to SFT data quality. MetaMath (Yu et al., ICLR 2024) introduced question bootstrapping — rewriting math problems via answer augmentation, rephrasing, and backward reasoning — and found a strong correlation (ρ = 0.97) between question diversity and model accuracy. MetaMath-7B achieved +11.5% over baseline on GSM8K.

Skywork-Math (Zeng et al., 2024) further demonstrated that scaling plain SFT on up to a million synthetic math examples enables even vanilla 7B models to achieve >82% accuracy on GSM8K. Their key finding: mathematical reasoning SFT benefits from both data diversity and scale, unlike general instruction-following where diminishing returns appear quickly.

Orca-Math (Mitra et al., 2024) took a different approach, using a multi-agent framework (AgentInstruct) to generate 200K diverse math word problems with varying difficulty levels. Critically, this work was specifically designed for **small language models**, making it directly relevant to our ~42M parameter picochat model.

### 2.3 Conversation Quality Scaling

UltraChat (Ding et al., 2023) generated 1.4M multi-turn dialogues using ChatGPT, covering topics like world knowledge, writing, and coding. The filtered UltraChat-200k subset applies truecasing corrections and removes robotic responses ("I do not have emotions"), producing cleaner training signal. This dataset complements SmolTalk by providing more diverse, higher-quality multi-turn conversations.

SmolTalk (HuggingFace, 2024) is specifically designed for smaller models, with simpler conversation structures. Using both SmolTalk and UltraChat provides a curriculum-like effect: SmolTalk teaches basic conversational patterns, while UltraChat adds more sophisticated multi-turn reasoning.

## 3. Original SFT Configuration

The nanochat SFT pipeline (`scripts/chat_sft.py`) uses the following data mixture:

| Dataset | Source | Rows | Purpose |
|---------|--------|------|---------|
| SmolTalk | `HuggingFaceTB/smol-smoltalk` | 460K | General conversational ability |
| Identity | Custom JSONL | 2K (2 epochs) | Nanochat personality/identity |
| MMLU | `cais/mmlu` (auxiliary_train) | 300K (3 epochs) | Multiple choice reasoning |
| GSM8K | `openai/gsm8k` (main) | 32K (4 epochs) | Math reasoning + tool use |
| SimpleSpelling | Generated | 200K | Basic word spelling |
| SpellingBee | Generated | 80K | Letter counting in words |
| **Total** | | **~1.07M** | |

**Justification of each component:**

- **SmolTalk** (460K): The backbone conversational dataset. Designed specifically for smaller models, it provides the majority of general instruction-following examples. Using the "smol" variant (vs. full SmolTalk) prevents the model from being overwhelmed with complex conversations beyond its capacity.

- **Identity conversations** (2K × 2 epochs): Synthetic conversations that impart nanochat's personality. Oversampled 2× to ensure the model reliably responds with its identity. A small but critical component — without it, the model defaults to generic assistant behavior.

- **MMLU** (100K × 3 epochs): Teaches structured multiple-choice reasoning across 57 academic subjects. 3× oversampling based on SFT sweeps (commit `1415fb7`) that found this ratio optimal for the data mixture balance. Uses the `auxiliary_train` subset (100K rows) rather than the full MMLU.

- **GSM8K** (8K × 4 epochs): Teaches mathematical reasoning with Python tool calls. The tool-call format (using `<|python_start|>`/`<|python_end|>` special tokens) is unique to GSM8K in the mixture. 4× oversampling compensates for the small dataset size relative to the rest of the mixture.

- **SimpleSpelling + SpellingBee** (280K combined): Teaches character-level awareness — spelling words letter-by-letter and counting specific letters. These address a known weakness of BPE-tokenized models that operate on subword tokens rather than characters.

**Training hyperparameters** (inherited from pretrained checkpoint unless overridden):
- Optimizer: MuonAdamW (Muon for matrix params, AdamW for embeddings)
- Weight decay: 0.0 (decayed to zero during pretraining)
- LR schedule: constant for 50% → linear warmdown to 0 for remaining 50%
- Initial LR fraction: 0.8× (slight reduction from base)
- Optimizer warm-start: Load momentum buffers from pretrained checkpoint, reset LRs
- Evaluation: ChatCORE every 200 steps, val BPB every 200 steps

## 4. New Dataset Selection

### 4.1 MetaMathQA

- **Source**: `meta-math/MetaMathQA` (395K rows, we use 50K)
- **Paper**: Yu et al., "MetaMath: Bootstrap Your Own Mathematical Questions for Large Language Models", ICLR 2024
- **Format**: Single-turn Q&A — `query` → user message, `response` → assistant message

**Why this dataset:** MetaMathQA directly targets GSM8K improvement through question diversity. The dataset applies four augmentation strategies to GSM8K and MATH training problems: (1) answer augmentation (AnsAug) generating multiple solution paths, (2) LLM-driven question rephrasing, (3) backward reasoning (FOBAR — Fill-in-the-blank with Backward Reasoning), and (4) single-variable solving (SV). The ablation study in the original paper found that merging all augmentation types outperforms any single strategy, and diversity correlates with accuracy at ρ = 0.97.

**Size choice (50K):** We subsample rather than using the full 395K to avoid overwhelming the mixture. MetaMath's own experiments show diminishing returns beyond 100K rows, and our model at ~42M parameters has limited capacity to absorb large datasets.

### 4.2 Orca-Math

- **Source**: `microsoft/orca-math-word-problems-200k` (200K rows, we use 50K)
- **Paper**: Mitra et al., "Orca-Math: Unlocking the potential of SLMs in Grade School Math", 2024
- **Format**: Single-turn Q&A — `question` → user message, `answer` → assistant message (with detailed step-by-step solutions)

**Why this dataset:** Orca-Math was specifically designed for **small language models**. While MetaMathQA was primarily validated on 7B-70B models, Orca-Math explicitly targets the SLM regime. The dataset was generated using a multi-agent framework (AgentInstruct) where generator, modifier, and verifier agents collaboratively create and refine problems at varying difficulty levels. The result: Orca-Math-7B achieved 86.81% on GSM8K, outperforming models 10× larger.

**Complementarity with MetaMathQA:** MetaMathQA augments *existing* GSM8K/MATH problems, while Orca-Math generates *novel* problems from scratch. This provides orthogonal training signal — augmented versions of known problems plus entirely new problem types.

### 4.3 UltraChat-200k

- **Source**: `HuggingFaceH4/ultrachat_200k` (208K SFT split, we use 100K)
- **Paper**: Ding et al., "Enhancing Chat Language Models by Scaling High-quality Instructional Conversations", 2023
- **Format**: Multi-turn conversations with alternating user/assistant messages — directly compatible with nanochat's conversation format

**Why this dataset:** UltraChat-200k was the SFT dataset behind Zephyr-7B-β, one of the strongest open 7B chat models at its release. The quality filtering process removes grammatical errors (truecasing ~5% of data) and robotic responses, producing cleaner training signal than raw ChatGPT generations. It covers a broad range of topics including world knowledge, creative writing, and analytical reasoning.

**Complementarity with SmolTalk:** SmolTalk is designed for small models with simpler conversation structures. UltraChat provides more sophisticated multi-turn interactions. Using both creates a difficulty spectrum: SmolTalk teaches basic conversational patterns, UltraChat teaches more complex reasoning chains and topic transitions.

## 5. Method

### 5.1 Experimental Setup

**Base model**: picochat (depth=8, ~42M parameters), pretrained with context extension curriculum (seq_len 512→2048, 5000 steps). Loaded from `base_checkpoints/picochat-ctx-s1` at step 5000.

**Two SFT configurations:**

| Configuration | Mixture | Total Rows |
|--------------|---------|------------|
| Baseline | SmolTalk + Identity + MMLU×3 + GSM8K×4 + Spelling | ~1.07M |
| Enhanced | Baseline + MetaMathQA 50K + Orca-Math 50K + UltraChat 100K | ~1.27M |

Both configurations use identical hyperparameters — only the data mixture differs. Training hyperparameters are inherited from the pretrained checkpoint.

**Compute**: Modal H100:1 (single GPU), ~16 device batch size.

### 5.2 Implementation

New task files follow the existing `tasks/` pattern:
- `tasks/metamathqa.py`: Loads from HuggingFace, maps `query`/`response` to user/assistant messages
- `tasks/orcamath.py`: Loads from HuggingFace, maps `question`/`answer` to user/assistant messages
- `tasks/ultrachat.py`: Loads from HuggingFace, validates message structure (role alternation)

The SFT script (`scripts/chat_sft.py`) was extended with three new CLI flags (`--metamathqa-size`, `--orcamath-size`, `--ultrachat-size`) that conditionally add datasets to the training mixture when set to values > 0. The default (0) preserves the original behavior.

## 7. Reproduction

### Run on Modal

```bash
# Full pipeline (baseline SFT + enhanced SFT + evals)
uv run modal run sft_modal.py

# Or individual stages
uv run modal run sft_modal.py::run_baseline_sft
uv run modal run sft_modal.py::run_enhanced_sft
uv run modal run sft_modal.py::run_sft_evals
```

### Run locally (requires pretrained checkpoint)

```bash
# Download identity conversations
curl -L -o $NANOCHAT_BASE_DIR/identity_conversations.jsonl \
    https://karpathy-public.s3.us-west-2.amazonaws.com/identity_conversations.jsonl

# Baseline SFT
torchrun --standalone --nproc_per_node=8 -m scripts.chat_sft -- \
    --device-batch-size=16 --run=sft-baseline

# Enhanced SFT
torchrun --standalone --nproc_per_node=8 -m scripts.chat_sft -- \
    --device-batch-size=16 --run=sft-enhanced \
    --metamathqa-size=50000 --orcamath-size=50000 --ultrachat-size=100000

# Evaluate
torchrun --standalone --nproc_per_node=8 -m scripts.chat_eval -- -i sft
```
